# 常见概念与面试题解析

> 本文档覆盖数据结构面试中的高频话题，按相关性组织。建议按顺序阅读，但每节也可独立查阅。

## 一、数组与链表

### 数组（Array）

**存储方式**：一块连续的内存空间，存储相同类型的元素。

| 维度 | 说明 |
|------|------|
| 随机访问 | O(1) —— 通过索引直接寻址 |
| 插入/删除 | O(n) —— 需移动后续元素腾出或填补空间 |
| 长度 | 固定，创建时指定，不可动态改变 |
| 内存密度 | 高——仅有数据，无额外指针开销 |
| 内存碎片化 | 大块连续分配可能导致碎片 |

### 链表（Linked List）

**存储方式**：由节点组成，每个节点包含值 + 指向后续节点的指针。

- **单向链表**：节点含 `next` 指针
- **双向链表**：节点含 `prev` + `next` 指针
- **循环链表**：尾节点指向头节点

| 维度 | 说明 |
|------|------|
| 随机访问 | O(n) —— 需从头遍历 |
| 插入/删除 | O(1) —— 只改指针（假设已持有目标节点引用） |
| 长度 | 动态增长，无需预定义 |
| 内存密度 | 低——每个节点额外存储指针开销 |
| 缓存友好性 | 差——节点分散在内存中 |

### 对比总结

| 场景 | 推荐选择 |
|------|---------|
| 频繁随机访问 | 数组 |
| 频繁插入/删除 | 链表 |
| 大小已知，较少扩容 | 数组 |
| 大小不确定，动态增减 | 链表 |
| 内存受限 | 数组（更紧凑） |
| 追求缓存命中率 | 数组（连续内存，CPU 预读友好） |

## 二、栈与队列

### 栈（Stack）

**核心特性**：后进先出（LIFO, Last-In-First-Out）

- 操作限定在**栈顶**（top）
- `push`：压入栈顶
- `pop`：弹出栈顶元素
- `top` / `peek`：查看栈顶元素

**复杂度**：所有操作 O(1)

**现实场景**：

| 场景 | 说明 |
|------|------|
| 函数调用栈 | 保存返回地址和局部变量，递归深度过大时栈溢出 |
| 浏览器的前进/后退 | 两个栈分别记录历史，后退=弹出，前进=压入 |
| 撤销操作（Undo） | 每次操作入栈，撤销时出栈 |
| 括号匹配 | 左括号入栈，右括号与栈顶匹配 |
| 逆波兰表达式求值 | 操作数入栈，遇运算符弹出计算后压回 |
| 深度优先搜索（DFS） | 显式或隐式使用栈管理待遍历节点 |

### 队列（Queue）

**核心特性**：先进先出（FIFO, First-In-First-Out）

- `enqueue`：从队尾入队
- `dequeue`：从队首出队
- `front`：查看队首元素

**复杂度**：所有操作 O(1)（链表或循环数组实现）

**变体**：

| 变体 | 特点 | 场景 |
|------|------|------|
| 普通队列 | FIFO | 任务调度、打印机排队 |
| 双端队列（Deque） | 两端均可插入/删除 | 滑动窗口极值 |
| 优先队列（Priority Queue） | 按优先级出队 | Dijkstra、任务优先级调度 |
| 阻塞队列 | 队列空/满时阻塞等待 | 生产者-消费者模式 |
| 循环队列 | 重用数组空间 | 固定大小的缓冲区 |

**现实场景**：

| 场景 | 说明 |
|------|------|
| 消息队列 | Kafka、RabbitMQ 等实现异步解耦 |
| 线程池任务队列 | 等待线程按提交顺序执行 |
| 打印机队列 | 按提交次序处理打印任务 |
| 广度优先搜索（BFS） | 显式使用队列管理待遍历节点 |
| 缓冲区管理 | IO 数据、音视频数据的分批处理 |

## 三、哈希表

### 什么是哈希

**哈希**（Hash）是将任意长度的输入数据通过**哈希函数**（Hash Function）映射为固定长度输出的过程。

**哈希函数的关键性质**：

| 性质 | 说明 |
|------|------|
| **确定性** | 相同输入始终得到相同哈希值 |
| **固定输出长度** | 无论输入多长，输出长度不变 |
| **雪崩效应** | 输入微小变化，输出显著不同 |
| **不可逆性** | 从哈希值无法反推原始输入（单向性） |
| **冲突必然性** | 不同输入**可能**产生相同哈希值（鸽巢原理） |

> 📌 **重要概念澄清**：**哈希不是加密**。哈希是单向映射，而加密是可逆的。不同的输入可能产生相同的哈希值，这称为**哈希冲突**（Hash Collision）——由鸽巢原理保证必然存在。

**常见哈希函数**：

| 函数 | 输出长度 | 用途 | 安全等级 |
|------|---------|------|---------|
| MD5 | 128 bit | 数据校验、文件指纹 | ❌ 已破解，存在碰撞攻击 |
| SHA-1 | 160 bit | 过去广泛用于完整性验证 | ❌ 已弃用 |
| SHA-256 | 256 bit | 通用安全哈希（HTTPS、证书） | ✅ |
| SHA-3 | 可选 | 新一代标准 | ✅ |
| Blake2 | 可选 | 高性能安全哈希 | ✅ |

**主要应用**：

- 哈希表（键值对存储和快速查找）
- 数据完整性校验（文件 MD5/SHA 对比）
- 密码学（密码哈希存储、数字签名）
- 分布式系统（一致性哈希确定节点位置）

### 哈希冲突的解决方式

#### ① 开放定址法（Open Addressing）

发生冲突时，在散列表中探测下一个空闲位置。

| 探测方式 | 公式 | 特点 |
|---------|------|------|
| 线性探测 | `h(k) + i` | 简单，但有**聚集**现象（primary clustering） |
| 二次探测 | `h(k) + i²` | 缓解聚集，但可能无法访问所有槽 |
| 双重哈希 | `h₁(k) + i·h₂(k)` | 分布最均匀，计算开销略大 |

- **优点**：不需要额外空间存储指针
- **缺点**：删除需使用**惰性删除**标记，负载因子高了性能骤降

#### ② 链地址法（Chaining）

每个槽位维护一个链表（或红黑树、数组），冲突元素插入链表末尾。

```
槽0 → [节点A] → [节点C]
槽1 → [节点B]
槽2 → NULL
```

| 优化策略 | 说明 |
|---------|------|
| 链表 → 红黑树 | Java 8+ 在链表长度 ≥ 8 时转为红黑树，退化时再转回链表 |
| 动态调整 | 负载因子过大时扩容 + rehash |
| 头插法 vs 尾插法 | 头插法更快但不保证插入顺序 |

- **优点**：实现简单，删除方便，支持高负载因子
- **缺点**：额外指针开销，最坏情况退化为单链表 O(n)

#### ③ 再哈希法 / 双重哈希（Double Hashing）

使用多个哈希函数，当 `h₁` 发生冲突时依次尝试 `h₂`、`h₃`……

```
索引 = h₁(key)             // 冲突
索引 = h₂(key)             // 冲突
索引 = h₃(key)             // 可用
```

> ⚠️ **术语澄清**：面试中常说的"再哈希"实际上指**双重哈希**（double hashing），这是**开放定址法**的一个变种。不要与哈希表扩容时全量**rehash**（重建哈希表）混淆。

- **优点**：分布均匀，减少聚集
- **缺点**：需保证多个哈希函数计算效率高，且相对独立

#### ④ 建立公共溢出区

冲突元素统一存入一个独立的**溢出表**中。

- 查找时先在主表中查找，未命中则到溢出区中查找
- 适用于冲突较少、溢出区不大的场景
- 溢出区可用更简单的数据结构（如链表）

#### 四种方法对比

| 方法 | 空间开销 | 删除操作 | 高负载因子表现 | 实现复杂度 |
|------|---------|---------|--------------|-----------|
| 开放定址法 | 低 | 需惰性删除 | ❌ 差 | 中 |
| 链地址法 | 中（指针） | 简单 | ✅ 好 | 低 |
| 双重哈希 | 低 | 需惰性删除 | ✅ 较好 | 高 |
| 公共溢出区 | 中（溢出区） | 简单 | ❌ 差 | 低 |

### Rehash 代价优化

**Rehash**：当负载因子（元素数 / 容量）超过阈值时，创建一个更大的哈希表，将所有元素重新插入。

**Rehash 的代价**：
- 需要 O(m) 时间遍历所有元素重新计算哈希
- 需要 O(new_cap) 空间分配新桶数组
- 期间对哈希表的正常操作有性能抖动

**优化策略**：

| 策略 | 说明 |
|------|------|
| **初始容量预估** | 预先估计数据量，分配足够大的初始容量，延缓 rehash |
| **合理负载因子** | 默认 0.75 是一个折中；内存充裕可用 0.5（更快），内存紧张可达 0.9（省空间） |
| **渐进式 rehash** | 每次插入/查找时**只迁移一部分**元素，分摊 O(m) 成本至多次操作（Redis dict 采用） |
| **分片（Sharding）** | 将哈希表拆分为多个独立子表，rehash 只在单个分片上发生 |
| **免 rehash 结构** | 使用**一致性哈希**（分布式）、**布谷鸟哈希**（Cuckoo Hashing）等替代方案 |

**Redis 渐进式 rehash 示意**：

```python
class Dict:
    def __init__(self):
        self.dict0 = []      # 主表（旧）
        self.dict1 = None    # 新表（rehash 目标）
        self.rehash_idx = 0  # 已迁移进度

    def get(self, key):
        # 每次查找时顺便迁移 rehash_idx 桶
        if self.dict1 is not None:
            self._migrate_bucket(self.dict0[self.rehash_idx])
            self.rehash_idx += 1
        # 先查新表，再查旧表
        ...
```

> 🔧 在 Python 中，`dict` 和 `set` 使用**开放定址法** + 线性探测；Java 的 `HashMap` 使用**链地址法**（链表 → 红黑树）。了解这些实现细节在面试和工程选型时都有用。

## 四、树

### 二叉树基础

**二叉树**（Binary Tree）：每个节点最多有两个子节点（左、右）。

**基础遍历**（递归/迭代）：

| 遍历方式 | 顺序 | 典型应用 |
|---------|------|---------|
| **前序**（Preorder） | 根 → 左 → 右 | 序列化二叉树、目录树展示 |
| **中序**（Inorder） | 左 → 根 → 右 | BST 输出有序序列 |
| **后序**（Postorder） | 左 → 右 → 根 | 释放树、计算目录大小 |
| **层序**（Level-order） | 逐层从左到右 | BFS、最短路径 |

**特殊二叉树**：

| 类型 | 定义 | 特点 |
|------|------|------|
| **满二叉树** | 所有内部节点都有 2 个子节点，叶子节点全在同一层 | 节点数 = 2ʰ⁺¹ − 1（h 为高度） |
| **完全二叉树** | 除最后一层外全满，最后一层节点尽量靠左 | 可用数组存储（堆的底层结构） |
| **二叉搜索树（BST）** | 左 ≤ 根 ≤ 右 | 中序有序，查找 O(h) h ∈ [log₂N, N] |
| **平衡二叉树（AVL）** | 任意节点左右子树高度差 ≤ 1 | 严格平衡，查找 O(logN)，插入删除开销稍大 |

### 红黑树

**红黑树**（Red-Black Tree）是一种近似平衡的二叉搜索树，通过颜色约束保证最长路径不超过最短路径的 2 倍。

#### 五条规则

1. 每个节点要么**红色**，要么**黑色**
2. **根节点**必须是黑色
3. 所有叶子节点（NIL）视为黑色
4. 红色节点的子节点必须是黑色（不允许连续的红色节点）
5. 任意节点到其所有后代叶子节点的路径上，黑色节点数相同（**黑高**相等）

#### 关键性质

| 性质 | 值 |
|------|----|
| 搜索时间复杂度 | O(logN) |
| 插入/删除时间复杂度 | O(logN) |
| 平衡程度 | 近似平衡（非严格） |
| 与 AVL 对比 | 红黑树插入/删除旋转更少，写操作更优 |

#### 插入操作要点

1. 新节点着色为**红色**（不破坏规则 5）
2. 检查父节点颜色
   - 父节点为黑色 → 无需调整
   - 父节点为红色 → 违反规则 4，需旋转 + 变色
3. 最终将根节点强制设为黑色

#### 删除操作要点

1. 若被删除节点有两个非空子节点，找到**后继节点**（右子树的最小值）替换
2. 删除后继节点（实际删除的节点至多有一个子节点）
3. 若被删除节点是黑色，则破坏规则 5，需修复

#### 红黑树 vs AVL 树

| 维度 | AVL | 红黑树 |
|------|-----|--------|
| 平衡程度 | 严格平衡 | 近似平衡 |
| 查找 | 更快（树高更低） | 略慢 |
| 插入/删除 | 旋转更多 | 旋转更少 |
| 使用场景 | 读多写少（数据库索引） | 读写均衡（`std::map`、Java `TreeMap`） |

> **实际应用**：Java `TreeMap`、`TreeSet`、C++ `std::map`、Linux 完全公平调度器、Nginx 定时器管理均基于红黑树。

## 五、排序算法比较

### 堆排序（Heap Sort）

```python
def heapify(arr, n, i):
    largest = i
    left = 2 * i + 1
    right = 2 * i + 2
    if left < n and arr[left] > arr[largest]:
        largest = left
    if right < n and arr[right] > arr[largest]:
        largest = right
    if largest != i:
        arr[i], arr[largest] = arr[largest], arr[i]
        heapify(arr, n, largest)

def heap_sort(arr):
    n = len(arr)
    # 建堆
    for i in range(n // 2 - 1, -1, -1):
        heapify(arr, n, i)
    # 逐个取出堆顶
    for i in range(n - 1, 0, -1):
        arr[0], arr[i] = arr[i], arr[0]
        heapify(arr, i, 0)
```

**核心机制**：
1. 将数组构建成**最大堆**（Max-Heap）
2. 每次取堆顶（最大值）与末尾元素交换，缩小堆的尺寸，重新调整
3. 重复直至所有元素有序

### 快速排序（Quick Sort）

```python
def quick_sort(arr, low=0, high=None):
    if high is None:
        high = len(arr) - 1
    if low < high:
        pi = partition(arr, low, high)
        quick_sort(arr, low, pi - 1)
        quick_sort(arr, pi + 1, high)

def partition(arr, low, high):
    pivot = arr[high]        # 选最右为基准
    i = low - 1
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[high] = arr[high], arr[i + 1]
    return i + 1
```

**核心机制**：
1. 选取**基准元素**（pivot），将数组分为**左小右大**两部分
2. 递归对左右子数组做同样操作
3. 分治，直到子数组长度为 1

### 对比

| 维度 | 堆排序 | 快速排序 |
|------|--------|---------|
| 平均时间复杂度 | O(nlogn) | O(nlogn) |
| 最坏时间复杂度 | O(nlogn) —— 稳定 | O(n²) —— 当已有序/逆序且选最值作 pivot |
| 最优时间复杂度 | O(nlogn) | O(nlogn) |
| 空间复杂度 | O(1) 原地排序 | O(logn) 递归栈 |
| 稳定性 | ❌ 不稳定 | ❌ 不稳定（取决于 partition 实现） |
| 缓存友好性 | ❌ 差（跳跃访问） | ✅ 好（顺序访问） |
| 实际工程表现 | 较慢（频繁的堆化） | 较快（cache 友好） |

> **工程实测**：在大部分数据集上，快排比堆排序快 2~3 倍，主要得益于 CPU 缓存的顺序访问模式。堆排序的"swap 堆顶 + heapify"过程产生大量随机内存访问。

### 如何选择

| 场景 | 推荐 |
|------|------|
| 常规排序，内存充足 | **快速排序**（实际最快） |
| 需要稳定性 | **归并排序** |
| 内存严格受限 | **堆排序**（原地、O(1) 空间） |
| 超大数组，外部排序 | **多路归并** |
| 小数组（N ≤ 16） | **插入排序**（常数级最快） |

## 附录：各节关联关系

```
哈希表 ← 一致性哈希用于分布式节点定位
    ↓
哈希冲突解决 → 链地址法（HashMap）与 开放定址法（Python dict）
    ↓
Rehash 优化 → 渐进式 rehash（Redis） → 布谷鸟哈希替代
    ↓
树结构 ← BST 退化 → 自平衡 AVL/红黑树
    ↓
红黑树 ← STL map / Java TreeMap 底层
    ↓
排序 ← 堆结构（完全二叉树） → 堆排序
    ↓
快速排序 ← 分治 → 递归（栈实现）
```
